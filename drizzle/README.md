# Migrations

This folder is managed by [drizzle-kit](https://orm.drizzle.team/kit-docs/overview).

## Status

`0000_messy_proteus.sql` was generated by `drizzle-kit generate`.
`0001_add_auth.sql` through `0004_daily_results.sql` were authored by hand
and registered in `meta/_journal.json` manually because some of them
(notably adding a composite PRIMARY KEY to an existing table, and the
schema dedup step in `0002`) need imperative SQL that `drizzle-kit`
won't emit cleanly.

The schema in `src/db/schema.ts` is the source of truth for the *current*
state — all four hand-rolled migrations have been applied to the live
schema there. The `meta/` snapshot JSON files, however, only cover
`0000_messy_proteus`.

## What this means in practice

- **Applying migrations is fine** — `npm run db:migrate` walks the
  journal entries in order and runs the `.sql` files.

- **Generating new migrations** with `npm run db:generate` will emit a
  no-op `0005_*.sql` the first time you run it, because drizzle-kit
  diffs against `meta/0000_snapshot.json` and sees auth + daily_results
  as additions.

  When that happens:

  1. Delete the no-op migration drizzle-kit just generated.
  2. Run `drizzle-kit generate --name=baseline` to snapshot the current
     schema state into `meta/`.
  3. Manually fold the new snapshot files into the journal alongside
     the hand-rolled entries.

  Or, the nuclear option: drop the dev DB, delete *all* of `drizzle/`,
  run `drizzle-kit generate` once to produce a clean baseline, then
  re-seed. That's only sensible before any real users exist.

## Migration list

| #    | Name                  | Why                                                                  |
| ---- | --------------------- | -------------------------------------------------------------------- |
| 0000 | `messy_proteus`       | Initial `cards` + `user_progress` tables (drizzle-generated).        |
